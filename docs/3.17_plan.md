# 2026-03-17 Plan Notes

## 关于 `deploy.sh`

当前 `deploy.sh` 的定位是：

- 手工触发的本地部署脚本
- 负责本地测试、从本地同步服务器、远端快速回归、重启服务、打印版本信息

它**不是**下面这些东西：

- 不是 CI
- 不是 CD
- 不是多环境发布系统
- 不是批量发布系统

## CI 是什么

CI = Continuous Integration，持续集成。

最通俗的理解：

- 你一 push 代码
- 系统自动跑测试 / 校验 / lint
- 尽早告诉你这次改动有没有把仓库搞坏

在本仓库里，如果后面要做 CI，第一步最适合做的是：

- 自动跑一组快速回归测试
- 不依赖真实企业微信
- 不依赖真实线上服务器
- 不强依赖外网 API

## CD 是什么

CD 一般有两种常见说法：

1. Continuous Delivery
   - 代码随时处于“可发布”状态
   - 但最终发不发，仍由人按一下按钮

2. Continuous Deployment
   - 代码一通过检查就自动发出去
   - 中间不再人工确认

对当前仓库来说：

- 现在还没有 CI
- 也没有 CD
- 当前是“人工测试 + 人工执行 deploy.sh + 人工核对版本”的模式

## 为什么当前 `deploy.sh` 不是 CI / CD

因为它具备的是“部署脚本”特征，而不是“平台自动化流水线”特征：

- 需要人手动执行
- 需要人交互选择测试模式
- 默认只面向一台服务器
- 没有和 GitHub push / PR 自动绑定
- 没有自动 gating 规则

## 什么是 Quick Regression

Quick Regression = 快速回归测试。

它不是一种特殊技术，而是一组“改动后最应该第一时间跑”的测试集合。

目标：

- 足够快
- 覆盖最核心逻辑
- 一旦挂了，通常说明主路径被改坏了

在这个仓库里，当前 quick regression 更像是：

```bash
PYTHONPATH=. pytest \
  tests/test_profile_state.py \
  tests/test_nodes_unit.py \
  tests/test_extractor_eval.py \
  tests/test_intent_guardrails.py \
  -q
```

为什么这叫 quick regression：

- 它比全量测试快
- 它能覆盖 state / router / nodes / eval 这些主干
- 很适合每次发布前先跑一次

它的本质不是“测得最全”，而是“最快发现主功能回归”

## 什么是 External Test

External test = 依赖真实外部系统的测试。

在这个仓库里，下面这些更接近 external：

- 真打 DeepSeek / OpenAI / Gemini API
- 真连 PostgreSQL / Redis
- 真打企业微信接口
- 真起服务后通过外部请求验证网页端

为什么先不把 external 纳入默认发布检查：

- 易受网络、密钥、配额、服务抖动影响
- 容易把“代码坏了”和“外部服务抖了”混在一起
- 当前阶段会降低发布稳定性

## 后续建议

当前先不动测试目录结构。

后续等节点 prompt 流水线测试补完后，再一起做：

1. 定义 quick regression 测试集合
2. 定义哪些测试属于 external
3. 再考虑是否调整目录结构或加 pytest markers
